# White Paper Scaffold: Building Research Software with AI Agents

## Working Title
"From Solo Developer to AI Swarm: How SimCash Was Built by Humans and Agents"

---

## 1. Abstract

This paper traces the development of SimCash, a payment system simulation platform, through 2,181 commits over 4 months. SimCash provides a unique case study in the evolution of AI-assisted software development: it began as a solo project with a human developer using Claude Code as a coding assistant, transitioned through a phase of intensive PR-based AI development, and culminated in a multi-agent architecture where autonomous AI agents (Nash, Stefan) built and deployed a production web platform with minimal human intervention. We document the methodology, tooling, and organizational patterns that emerged, and argue that persistent AI agents with identity, memory, and specialization represent a qualitatively different development paradigm from traditional AI coding assistants.

---

## 2. The Project: What SimCash Is

### 2.1 Domain
SimCash simulates Real-Time Gross Settlement (RTGS) payment systems — the infrastructure through which banks settle high-value payments. Banks must balance liquidity costs against delay penalties, a strategic coordination problem studied by central banking researchers (Castro et al., Desai & Aldasoro).

### 2.2 Architecture
- **Rust simulation engine:** Deterministic, event-sourced, with RTGS + LSM settlement
- **Python API layer:** FastAPI, DuckDB persistence, LLM integration via PydanticAI
- **React/TypeScript web platform:** Interactive experiment runner with real-time WebSocket streaming
- **Decision tree policies:** JSON DSL evaluated by the engine, generated by LLMs

### 2.3 Why SimCash Matters for This Paper
SimCash is not trivial software. It implements real financial system mechanics (TARGET2 compliance, bootstrap statistical evaluation, multi-agent game theory). The fact that AI agents could build, deploy, and improve this system — including running 130+ research experiments — makes it a compelling demonstration of AI-agent capabilities.

## 3. Development Timeline: The Eras of SimCash

### 3.1 Era 1: The Sprint (Oct 27 – Nov 3, 2025)
- 125 commits in 8 days
- Human directing Claude Code, committing under human identity
- Entire Rust engine built from scratch
- Grand Plan as founding document

### 3.2 Era 2: Claude Code at Scale (Nov 4 – Dec 22, 2025)
- ~1,700 commits over 7 weeks
- PR-based development (420+ PRs)
- Claude creates branches, human merges
- Research pivot: BIS models, Castro replication, LLM optimization
- Paper generation system

### 3.3 Era 3: The Multi-Agent Era (Feb 17 – Feb 28, 2026)
- ~475 commits in 12 days
- Nash agent builds entire web platform
- Stefan agent runs experiments
- Human makes 1 commit
- Production deployment to GCP

### 3.4 The Gap
8 weeks of silence between Eras 2 and 3. What happened? OpenClaw platform development, agent architecture design.

## 4. Methodology Evolution

### 4.1 Phase-Based Planning → AI Implementation
The Grand Plan specified phases; Claude Code executed them. Human wrote "what," AI wrote "how."

### 4.2 The Claude Code Workflow
Branch → implement with TDD → PR → human merge. Session-based, no persistent identity.

### 4.3 Handover Prompts: The Bridge
Versioned context documents (v2-v5) for briefing new Claude Code sessions. Proto-agent-memory.

### 4.4 The Multi-Agent Paradigm
Persistent identity, memory files, proactive behavior, specialized roles. From "AI assistant" to "AI team member."

### 4.5 The Human Role Transformation
```
Era 1: Human = planner + quality controller (commits directly)
Era 2: Human = director + reviewer (merges PRs)
Era 3: Human = vision holder + occasional reviewer (1 commit in 12 days)
```

## 5. Key Architectural Decisions

### 5.1 Decisions Made Before Any Code
The Grand Plan (commit #1) specified: Rust core, Python API, PyO3 FFI, decision tree policies, LLM optimization. All survived to production.

### 5.2 The Replay Identity Quest
3 weeks of intense effort to achieve byte-identical simulation replay. This decision (event sourcing) enabled bootstrap evaluation and the web platform.

### 5.3 YAML-Only Experiments
19 refactoring phases (Dec 8-15) to eliminate custom Python code from experiments. Result: purely declarative experiment definitions.

### 5.4 Bootstrap Evaluation
Statistical paired comparison of policy changes. Multiple iterations from zero-delta bugs to production-grade evaluation.

## 6. The Multi-Agent Development Model

### 6.1 Agent Specialization
- **Nash:** Full-stack engineer. Built web platform, deployed to production, ran playtests.
- **Stefan:** Experiment specialist. Ran 130+ experiments, analyzed results, wrote paper sections.
- **Dennis:** Architectural reviewer. Provided feedback, identified bugs (not a committing agent).

### 6.2 Agent Infrastructure
- **OpenClaw:** Agent runtime platform
- **SOUL.md:** Agent identity and personality
- **MEMORY.md:** Long-term memory across sessions
- **HEARTBEAT.md:** Proactive task list
- **TOOLS.md:** Tool configuration
- **Subagents:** For parallelized subtasks

### 6.3 Communication Patterns
- Agents communicate via Telegram (with human)
- Handover documents between agents
- Commit messages as implicit coordination
- Feature request documents as shared specs

### 6.4 Quality Without PRs
In the multi-agent era, there are no PRs. Nash commits directly. Quality assurance comes from:
- TDD (tests written before code)
- Playtesting via browser automation
- Self-review patterns
- Human spot-checks
- Agent peer review (Dennis reviews Nash)

## 7. Findings

### 7.1 Quantitative
- **Commit velocity:** Similar across eras (~50-90 commits/peak day)
- **Agent contribution:** 99% of Era 3 commits by AI agents
- **Feature velocity:** Web platform from zero to production in 12 days
- **Research output:** 130+ experiments, 2 papers, 111+ linked data points

### 7.2 Qualitative
- **Persistent identity matters:** Nash's continuity across sessions enabled coherent multi-day feature arcs
- **Specialization emerges naturally:** Nash and Stefan developed distinct roles without explicit assignment
- **Plans are disposable:** Documents created for implementation, deleted after completion
- **The human becomes strategic:** As AI capability grew, human input shifted from tactical to strategic

## 8. Implications

### 8.1 For Software Development
The solo developer → Claude Code → multi-agent progression suggests a natural evolution. As AI agents gain persistent identity and memory, the development paradigm shifts from "AI-assisted coding" to "AI team management."

### 8.2 For Research Software
SimCash demonstrates that AI agents can not only build research software but also run experiments and contribute to papers. The YAML-only experiment architecture enables non-programmers (or other AI agents) to define and execute research.

### 8.3 For AI Agent Design
Key design patterns that emerged:
- **Identity files** (SOUL.md) give agents coherent behavior
- **Memory files** enable multi-session continuity
- **Heartbeats** enable proactive behavior
- **Subagents** enable parallel work
- **Specialization** is more effective than general-purpose agents

### 8.4 For the Future
The SimCash trajectory suggests that software projects may increasingly be defined by a small number of humans setting vision and constraints, with AI agent teams handling implementation, testing, deployment, experimentation, and analysis. The "developer" role may evolve into a "development architect" role — designing systems of AI agents rather than writing code directly.
